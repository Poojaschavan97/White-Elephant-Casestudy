# White Elephant – Hotel Booking System Case Study

## 🧩 Problem Statement

The White Elephant is a 23-room hotel aiming to increase occupancy to 90%.  
The hotel currently uses a basic, undocumented booking system inherited from previous owners. 

Challenges identified:

- No online booking capability  
- Limited support for corporate bookings  
- No structured maintenance tracking  
- Lack of management reporting  
- Poor visibility of booking data  
- Manual fault recording process  

Alan (Finance & Corporate Director) and Eric (Operations Manager) require a modern booking system with website integration to improve efficiency, increase corporate bookings, and support business growth.

---

## 📌 Project Overview

The objective of this project is to analyse, validate, and refine requirements for a new hotel booking system that will:

- Allow guests to view and book rooms online  
- Support corporate bookings (multiple rooms per transaction)  
- Process credit card payments securely  
- Generate management and operational reports  
- Track and report maintenance issues  
- Improve usability for reception staff  

---

## 🎯 Business Goals

- Achieve 90% occupancy rate  
- Increase corporate bookings to 40%  
- Improve operational efficiency  
- Provide reporting for business decision-making  
- Replace outdated legacy system  

---

## 🔍 Activities Performed

### 1️⃣ Stakeholder Identification & Analysis
Key stakeholders involved:

- Eric (Operations Manager)
- Alan (Director – Finance & Corporate Clients)
- Reception Staff
- Hotel Manager
- Corporate Guests
- Maintenance Engineer

---

### 2️⃣ Requirement Analysis

- Identified Functional Requirements (e.g., view rooms, book rooms, invoice corporate clients)
- Identified Non-Functional Requirements (performance, usability, security)
- Highlighted requirement conflicts (speed vs security)
- Identified ambiguous requirements (e.g., “system shall be easy to use”)
- Converted ambiguous requirements into SMART format

---

### 3️⃣ Requirement Catalogue Creation

Documented structured requirement entries including:

- R1: View Available Rooms  
- R2: Corporate Multiple Room Booking  

Each entry includes:
- Description  
- Source  
- Priority  
- Rationale  
- Associated NFRs  
- Related Requirements  

---

### 4️⃣ UML & Class Diagram Review

Analysed provided Class Diagram to evaluate:

- Room attributes (beds, facilities, size)
- Corporate booking capability
- Fault tracking capability
- Invoice-booking relationships
- Missing attributes in Booking entity

Identified structural limitations and improvement areas.

## ⚙️ Key Non-Functional Requirements

- Booking confirmation within 30 seconds (Performance)
- System must be easy to use (Usability)
- Secure system against hackers (Security)
- Reliable email confirmation system
